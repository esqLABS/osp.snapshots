# Parameter-promotion candidates

This is a development-time analysis artifact for issue #158. It is not shipped in the built package (the whole `dev/` folder is `.Rbuildignore`d) and it adds no exported function, no data object, and no runtime dependency. It is evidence, not an implementation.

**Implementing any promoted argument is deferred to per-factory follow-up issues, #157 first (for `create_protocol()`), then one issue per remaining factory.** Nothing here changes a factory signature, `create_parameter()`, or the `parameters = list(...)` argument; those stay exactly as they are today. A reader must not mistake the recommendation tables below for an implemented change. Each table is a starting point a follow-up can implement directly, and each follow-up owns the final decision when it lands.

The motivation: several `create_*()` factories accept a free-form `parameters = list(create_parameter(name = "...", ...))` argument whose `name` strings must match PK-Sim's internal parameter names exactly. They are neither validated nor discoverable from the signature. The package has already adopted a better pattern twice, in `create_output_interval(name, start_time, end_time, resolution)` and `create_event_selection(name, start_time)`, which expose plain unit-aware scalars with sensible default units and build the raw `Parameters` entries internally. This analysis provides a data-driven, per-factory ranking of which parameters are common and important enough to promote that way.

## Methodology

**Source set (D1).** The crawl targets PK-Sim full-project snapshot JSON files published in the `Open-Systems-Pharmacology` GitHub organisation, not the compound-only `osp_models()` templates (templates carry no protocols or simulations, so they cannot inform the protocol/schema/event/population factories). The discovery rule enumerated the non-archived public org repos whose name matches the model/DDI convention (suffix `-Model` or `-DDI`, prefix `Example_`, or the `Pregnancy-Models` repo): 231 such repos at crawl time. A file counts as a project snapshot only if its parsed top-level object contains at least one in-scope parameter-carrying section (`Compounds`, `Protocols`, `Events`, `ExpressionProfiles`, `Populations`).

**Selection (recorded subset, spec Non-goal 3).** Exhaustive crawling of all 231 repos is not required; reproducibility rests on the recorded manifest, not on completeness. The crawl selected all 103 `-Model` repos, all 7 `Example_`/`Pregnancy-Models` repos, and a deterministic 1-in-4 sample of the 121 `-DDI` repos (sorted by name, every fourth), giving 141 candidate repos. For each, the top-level project snapshot was resolved by trying the conventional `<RepoName>.json` and, when absent, the first root-level `*.json` in the pinned tree that parses as a project snapshot (repo naming varies: `7E3.json`, `Compound.json`, `Atomoxetine-model.json`). Of the 141 candidates, 28 have no root-level project snapshot (their snapshots live in subfolders, which is why all 7 `Example_`/`Pregnancy` repos dropped out, or they are code-only repos) and contributed nothing. The remaining **113 project snapshots (85 `-Model`, 28 `-DDI`) are exactly the files read**, listed in `source-manifest.csv`.

**Pinning (D2).** Each crawled file is recorded with its repository, resolved default branch, exact commit SHA at crawl time, in-repo path, and a SHA-pinned `raw.githubusercontent.com` URL (never a branch name or `latest`), so the counts are reproducible against the identical bytes even as upstream repos change. The full pinned list is `source-manifest.csv`.

**Tally (FR-002, FR-003, FR-005).** For each in-scope building-block kind, the crawl visits every block of that kind in every snapshot and records, per distinct parameter identifier (`Name`, or `Path` for the localized expression-profile parameters), the total occurrence count, the set of distinct observed units, the number of distinct source models it appears in (`n_models`, so cross-model breadth is distinguishable from repetition within one model), and the contributing source models. The raw per-parameter tally is `crawl-tally.csv` (authoritative) and `crawl-tally.md` (a human-readable view). Snapshots missing a section contribute nothing to that factory and do not abort the crawl (FR-006): of the 113 snapshots, compounds appear in 113, protocols in 113, expression profiles in 69, events in 23, and populations in 16.

**Promotion threshold (D5).** A parameter is a promotion candidate when it satisfies at least one of: **(a)** it appears in a clear majority of the crawled blocks of its kind, operationalised here as present on **>= 5 distinct source models** (and, where a kind has few blocks, a majority share of them); or **(b)** it maps to a canonical PK-Sim / OSPSuite.Core engine constant that PK-Sim treats as a core setting for that block even if it is sparse in the crawl (for example a dose-family member). Every candidate row states which of (a)/(b)/both it rests on. A factory with no parameter clearing the threshold records "No promotion candidates" explicitly with the counts that led there.

**Engine cross-check and unit validation (FR-009, FR-010, FR-011).** For each candidate, the canonical PK-Sim parameter name is the exact `Name`/`Path` string observed in the JSON, cited by value in prose (no external file path or external identifier). The proposed `default_unit` is validated against its stated `dimension` using the same source of truth the package uses at runtime, `validate_unit()` (which wraps `ospsuite::getUnitsForDimension()` / `ospsuite::ospUnits`); a candidate whose default unit is not valid for its dimension is not emitted. A reviewer can spot-check any row with a one-line `osp.snapshots::validate_unit("<default_unit>", "<dimension>")` call and get `TRUE` with no error. Where a parameter's observed units span multiple members of a family, the family-resolution rule is stated as a mapping from unit dimension to family member (see the dose family in the protocol and schema-item sections).

**FR-007 protocol split (Simple vs Advanced).** A protocol object with an `ApplicationType` field is a Simple protocol; its `Parameters` count toward `create_protocol()` only. A protocol object with `Schemas` (and no `ApplicationType`) is an Advanced protocol; its `Schemas[].Parameters` count toward `create_schema()` and its `Schemas[].SchemaItems[].Parameters` count toward `create_schema_item()`. The three levels are reported separately and never merged. In the corpus the split is clean: 901 Simple protocols, 625 Advanced protocols, and no protocol carrying both. Observed `ApplicationType` values (e.g. `Intravenous`) are recorded verbatim and are not validated against the factory's enum; reconciling that enum drift is a follow-up concern.

**FR-008 compound split (loose parameters vs alternatives / pKa / processes).** Only the loose `Compounds[].Parameters` list counts toward `create_compound()` promotion. The physicochemical properties (`Lipophilicity`, `FractionUnbound`, `Solubility`, `IntestinalPermeability`, `Permeability`) are each a list of alternatives with their own nested `Parameters`; `PkaTypes` and `Processes` are separate again. Those live in dedicated fields, not in the loose list, and process-level parameters count toward `create_process()`, not `create_compound()`. The compound section reports the excluded-location counts as context so a follow-up does not conflate physicochemical-property setters with loose parameters.

## Per-factory recommendations

### create_compound()

All 113 crawled snapshots carried at least one compound, so this factory's tally rests on the full corpus. The FR-008 boundary is the crucial point here: only the loose `Compounds[].Parameters` list (the target of `create_compound(parameters=)`) is counted as a promotion candidate. The physicochemical properties are set through dedicated fields, not through loose parameters, and each of those fields is a list of alternatives carrying its own nested `Parameters`. Those excluded locations are in fact the most common compound parameters in the corpus (as context, not candidates): `Lipophilicity` (331 occurrences, 113 models), `Fraction unbound (plasma, reference value)` (298 occurrences, 113 models), `Solubility at reference pH` (357 occurrences, 112 models), `Reference pH` (351 occurrences, 112 models), pKa entries (356 occurrences, 101 models), and `Specific intestinal permeability (transcellular)` (186 occurrences, 88 models). A follow-up must not conflate these physicochemical-property setters with the loose `parameters` argument; they belong to dedicated arguments of their own if ever promoted, and are out of scope for `create_compound(parameters=)`.

Within the loose `Compounds[].Parameters` list, one parameter clears the D5 threshold as a unit-bearing promotion candidate:

| argument | pksim_parameter | default_unit | dimension | occurrences | n_models | evidence |
|---|---|---|---|---|---|---|
| `molecular_weight` | `Molecular weight` | `g/mol` | Molecular weight | 265 | 113 | D5(a): present on all 113 models (100% breadth). Maps to the engine constant `Molecular weight`; the only loose parameter observed with a single consistent dimension (`g/mol`). |

The remaining loose parameters are dimensionless flags or sparse: `Cl` (79 occurrences, 37 models, no unit) and `F` (38 occurrences, 27 models, no unit) are dimensionless fractions/clearance flags carrying no unit to validate, so they are weak unit-promotion candidates and are left to the follow-up's judgment rather than promoted here; `Kd (FcRn) in endosomal space` (9 occurrences, 8 models, `µmol/l`), `Enable supersaturation` (8 occurrences, 5 models, flag), `Radius (solute)` (8 occurrences, 7 models), `Density (drug)` (7 occurrences, 4 models), and `Treat precipitated drug as` (7 occurrences, 4 models) are below the frequency bar or are boolean/enumeration flags. They are recorded in the raw tally for traceability but do not clear the threshold as scalar unit-promotion candidates.

### create_event()

Only 23 of the 113 crawled snapshots carried an `Events` section, and the events observed are feeding/meal definitions (the application factory itself is dominated by protocols, not events). Four distinct event parameters appear in the corpus, and only one clears the D5 breadth bar:

| argument | pksim_parameter | default_unit | dimension | occurrences | n_models | evidence |
|---|---|---|---|---|---|---|
| `meal_energy_content` | `Meal energy content` | `kcal` | Energy | 14 | 11 | D5(a): on 11 distinct models (>= 5 models), the only event parameter clearing the breadth bar. Maps to the engine constant `Meal energy content`; `kcal` validated against dimension `Energy`. |

The other three event parameters are below the >= 5-models bar and are recorded in the raw tally for traceability but not promoted: `Meal fraction solid` (3 occurrences, 3 models, dimensionless), `Gallbladder emptying lag time` (2 occurrences, 1 model, `min`, dimension Time), and `Meal volume` (1 occurrence, 1 model, `l`, dimension Volume). A follow-up may still choose to promote `meal_fraction_solid`, `meal_volume`, and a `gallbladder_emptying_lag_time` alongside `meal_energy_content` to make a coherent meal-definition signature, since they are the canonical companions of the same feeding event even though sparse in this corpus (D5(b) rationale for a coherent family); their units (`min` -> Time, `l` -> Volume) validate, but on frequency alone only `Meal energy content` qualifies.

### create_expression_profile()

69 of the 113 crawled snapshots carried an `ExpressionProfiles` section. The identifier for an expression-profile parameter is a localized `Path` (`LocalizedParameter` uses `Path`, not `Name`), and the tally found 1285 distinct paths. The distribution makes the outcome clear: 356 paths (27.7%) appear on only a single model, and the paths that recur are per-molecule and per-organ constructs, not reusable scalars. The three most widespread paths are `CYP3A4|Reference concentration`, `CYP3A4|t1/2 (intestine)`, and `CYP3A4|t1/2 (liver)` (each 139 occurrences on 56 models), and the single most common leaf is `Relative expression`, which is carried by 1062 distinct paths (one per organ/compartment localization such as `Organism|Liver|Pericentral|Intracellular|CYP3A4|Relative expression`). The full set of distinct leaf names is only eight (`Relative expression`, `Reference concentration`, `t1/2 (intestine)`, `t1/2 (liver)`, `Relative expression in blood cells`, `Relative expression in plasma`, `Fraction expressed on plasma-side membrane of vascular endothelium`, `Ontogeny factor GI`), but each leaf is bound to a specific molecule and organ path, so it is not a single scalar the caller could set once.

**No promotion candidates.** The parameter identity here is a per-molecule, per-organ localized `Path`, not a reusable unit-bearing scalar. A leaf such as `Relative expression` recurs across 1062 different paths precisely because it is instantiated once per organ/compartment; promoting it to a single plain argument would collapse a structural hierarchy that only makes sense per localization. The right ergonomic improvement for this factory (a molecule/organ/value builder) is structural rather than a flat scalar argument, so it does not fit the FR-012 promotion model and is deferred wholesale to the factory's follow-up. Supporting counts: 1285 distinct paths, 458 on >= 5 models but all of them per-organ localizations of the same eight leaves, maximum breadth 56 models for the CYP3A4 reference-concentration/half-life leaves.

### create_population()

16 of the 113 crawled snapshots carried a `Populations` section (populations are the least common in-scope block, as most published models are single-individual). The tally distinguishes two kinds of population parameter: the scalar demographic `ParameterRange` entries in `Populations[].Settings` (fixed keys `Age`, `Weight`, `Height`, `GestationalAge`, `BMI`, each carrying `Min`/`Max`/`Unit`) and the per-molecule `AdvancedParameters` (enzyme/transporter `Reference concentration` and `t1/2` values, e.g. `CYP3A4|Reference concentration` on 13 models). The demographic ranges are reusable scalars and are the promotion candidates; the advanced enzyme parameters are per-molecule localized values (the same structural situation as expression profiles) and are recorded in the raw tally but not promoted as flat scalar arguments.

| argument | pksim_parameter | default_unit | dimension | occurrences | n_models | evidence |
|---|---|---|---|---|---|---|
| `age` | `Age` | `year(s)` | Age in years | 73 | 16 | D5(a): on all 16 population-bearing models; the most common demographic range. Maps to the `Age` `ParameterRange`; observed units `day(s)`/`month(s)`/`year(s)` all in the age/time family, canonical default `year(s)`. |
| `weight` | `Weight` | `kg` | Mass | 38 | 11 | D5(a): on 11 of 16 models (>= 5 models, majority). Maps to the `Weight` `ParameterRange`; `kg` validated against dimension `Mass`. |
| `height` | `Height` | `cm` | Length | 16 | 7 | D5(a): on 7 models (>= 5 models). Maps to the `Height` `ParameterRange`; `cm` validated against dimension `Length`. |
| `bmi` | `BMI` | `kg/m²` | BMI | 29 | 7 | D5(a): on 7 models (>= 5 models). Maps to the `BMI` `ParameterRange`; `kg/m²` validated against dimension `BMI`. |

`GestationalAge` did not appear in this corpus (the pregnancy models publish their snapshots in subfolders and were not among the crawled files), but it is the fifth canonical demographic range and a follow-up should include it alongside the four above for completeness (D5(b): canonical range, low frequency here). The per-molecule advanced parameters (`CYP3A4|Reference concentration`, 57 occurrences / 13 models; `CYP3A4|t1/2 (intestine)` and `CYP3A4|t1/2 (liver)`, 47 occurrences / 12 models each; and similar CYP1A2/CYP2D6/CYP2C8/ABCB1/UGT1A4 entries) are frequent but per-molecule localized, so they belong to a structured expression-variability sub-API rather than a flat scalar argument, and are deferred with the same rationale as `create_expression_profile()`.

### create_process()

107 of the 113 crawled snapshots carried compound processes (metabolism, transport, and clearance). This is the richest factory after the protocol family: 40 distinct process parameters were tallied under `Compounds[].Processes[].Parameters`, and many clear the D5 breadth bar comfortably. The candidates below are the process parameters that are both frequent and carry a single coherent dimension. Enzyme-kinetics parameters dominate: `Km` (662 occurrences, 82 models) and `kcat` (546 occurrences, 70 models) are the Michaelis-Menten pair, `Ki` (258 occurrences, 49 models) is the inhibition constant, and `Specific clearance` and `GFR fraction` are the common clearance descriptors.

| argument | pksim_parameter | default_unit | dimension | occurrences | n_models | evidence |
|---|---|---|---|---|---|---|
| `km` | `Km` | `µmol/l` | Concentration (molar) | 662 | 82 | D5(a): 82 of 107 models (majority, >= 5). Maps to the engine constant `Km` (Michaelis-Menten). Observed units `µM`/`µmol/l`/`mM`/`nmol/l` are all molar-concentration variants; canonical default `µmol/l`. |
| `kcat` | `kcat` | `1/min` | Inversed time | 546 | 70 | D5(a): 70 of 107 models. Maps to the engine constant `kcat` (turnover number); single observed unit `1/min` (dimension Inversed time). |
| `ki` | `Ki` | `µmol/l` | Concentration (molar) | 258 | 49 | D5(a): 49 models (>= 5). Maps to the engine constant `Ki` (inhibition constant); observed units `nmol/l`/`µmol/l` are molar variants, canonical default `µmol/l`. |
| `specific_clearance` | `Specific clearance` | `1/min` | Inversed time | 114 | 50 | D5(a): 50 models (>= 5). Maps to the engine constant `Specific clearance`; observed `1/h`/`1/min` both Inversed time, canonical default `1/min`. |
| `lipophilicity` | `Lipophilicity (experiment)` | `Log Units` | Log Units | 70 | 42 | D5(a): 42 models (>= 5). Maps to the engine constant `Lipophilicity (experiment)`; single observed unit `Log Units`. |

Several further parameters clear the >= 5-models bar but carry a dimension whose default unit needs a follow-up decision, so they are recorded in the raw tally rather than fixed here: `GFR fraction` (203 occurrences, 91 models, dimensionless) and `Fraction unbound (experiment)` (97 occurrences, 49 models, dimensionless) are dimensionless fractions (no unit to validate); `In vitro Vmax for liver microsomes` (333 occurrences, 49 models, dimension Vmax per mass protein), `In vitro Vmax/recombinant enzyme` (173 occurrences, 40 models, dimension Vmax per recombinant enzyme), and `Content of CYP proteins in liver microsomes` (207 occurrences, 31 models, dimension Abundance per mass protein) are high-frequency but each spans two microsome/enzyme-normalised units, so their canonical default is left to the follow-up to pick per the FR-011 multi-unit rule; `Vmax` (113 occurrences, 41 models), `Plasma clearance` (96 occurrences, 48 models, dimension Flow per weight), `Transporter concentration` (79 occurrences, 33 models, molar), `EC50`/`Emax` (81 occurrences, 15 models each) are the induction pair. All are in the raw tally with their distinct units for the follow-up to promote.

### create_protocol()

98 of the 113 crawled snapshots carried Simple protocols. Per the FR-007 split, this factory counts only Simple protocols (a protocol object with an `ApplicationType` field and no `Schemas`); the crawl found 901 such Simple protocols against 625 Advanced ones, and no protocol carried both, so the split on `ApplicationType` presence is unambiguous. Five distinct parameters appear on Simple protocols, and the top four clear the D5 breadth bar. This is the family issue #157 targets first.

| argument | pksim_parameter | default_unit | dimension | occurrences | n_models | evidence |
|---|---|---|---|---|---|---|
| `dose` | `InputDose` | `mg` | Mass | 901 | 98 | D5(a)+(b): on all 98 Simple-protocol models (100% breadth) and the core dosing constant. Maps to the dose family via unit dimension (see resolution rule below): `mg` -> `Dose`/`InputDose` (Mass), `mg/kg` -> `DosePerBodyWeight` (Dose per body weight), `mg/m²` -> `DosePerBodySurfaceArea` (Dose per body surface area). All three observed in the corpus on the single JSON parameter `InputDose`. |
| `start_time` | `Start time` | `h` | Time | 900 | 98 | D5(a): on all 98 models. Maps to the engine constant `Start time`; single dimension Time (`h`). Already the pattern used by `create_event_selection(start_time)`. |
| `volume_of_water_per_body_weight` | `Volume of water/body weight` | `ml/kg` | Volume per body weight | 596 | 84 | D5(a): 84 of 98 models (majority). Maps to the engine constant `Volume of water/body weight`; single observed unit `ml/kg` (dimension Volume per body weight). |
| `infusion_time` | `Infusion time` | `h` | Time | 235 | 54 | D5(a): 54 models (>= 5). Maps to the engine constant `Infusion time`; observed `h`/`min`/`s` all Time, canonical default `h` (consistent with `create_output_interval` time defaults). |

`End time` (127 occurrences, 37 models, dimension Time) also clears the >= 5-models bar and is recorded in the raw tally; a follow-up may promote it as `end_time` alongside the four above.

**Dose family and unit-to-family resolution rule (FR-009, FR-011, AC-4).** PK-Sim stores the applied dose as a single JSON parameter named `InputDose`, whose *unit dimension* selects which canonical engine dose-family member the value belongs to. A follow-up implementing `dose` must resolve the family from the unit dimension of the supplied `dose_unit`, verbatim:

- a mass unit (`mg`, dimension `Mass`) -> `Dose` / `InputDose` (absolute dose);
- a body-weight-normalised unit (`mg/kg`, dimension `Dose per body weight`) -> `DosePerBodyWeight`;
- a body-surface-area-normalised unit (`mg/m²`, dimension `Dose per body surface area`) -> `DosePerBodySurfaceArea`.

All three dimensions and their default units validated against `validate_unit()`. The fourth canonical family member, an amount-based `Dose` in `mol`/`mmol` (dimension `Amount`), was not observed in this corpus but is part of the same family and is covered by the same dimension-driven rule (D5(b): canonical constant, low frequency). The repetition parameters `NumberOfRepetitions` and `TimeBetweenRepetitions` do not occur on Simple protocols in the corpus; they live at the schema level and are covered by `create_schema()`.

### create_schema()

85 of the 113 crawled snapshots carried Advanced protocols with `Schemas`. Per FR-007 this factory counts only the schema-level `Protocols[].Schemas[].Parameters` (not the deeper `SchemaItems[].Parameters`, which belong to `create_schema_item()`, and not Simple-protocol `Parameters`, which belong to `create_protocol()`). The crawl found exactly three distinct schema-level parameters, and all three clear the D5 breadth bar. The schema level is where the repetition family lives.

| argument | pksim_parameter | default_unit | dimension | occurrences | n_models | evidence |
|---|---|---|---|---|---|---|
| `number_of_repetitions` | `NumberOfRepetitions` | `(none)` | (dimensionless count) | 809 | 85 | D5(a)+(b): on all 85 schema-bearing models (100% breadth) and a core repetition constant. Maps to the engine constant `NumberOfRepetitions`; a dimensionless integer count, so it carries no unit to validate. |
| `time_between_repetitions` | `TimeBetweenRepetitions` | `h` | Time | 809 | 85 | D5(a)+(b): on all 85 models and a core repetition constant. Maps to the engine constant `TimeBetweenRepetitions`; observed `h`/`min`/`day(s)` all Time, canonical default `h`. |
| `start_time` | `Start time` | `h` | Time | 805 | 85 | D5(a): on all 85 models. Maps to the engine constant `Start time`; observed `h`/`min`/`day(s)` all Time, canonical default `h`. |

**Repetition family (FR-009, AC-4).** The two repetition parameters that #157/#158 call out, `NumberOfRepetitions` and `TimeBetweenRepetitions`, are owned by this factory: they are schema-level constants describing how a dosing schema repeats. `NumberOfRepetitions` is a dimensionless integer (a count of repetitions) and so carries no unit to validate; `TimeBetweenRepetitions` is a Time value whose default unit `h` validates against dimension `Time`. Both appear on every schema-bearing model in the corpus, so they are unambiguous promotions. The dose family (`InputDose`/`Dose`/`DosePerBodyWeight`/`DosePerBodySurfaceArea`) does not live at the schema level; it lives one level deeper on the schema items and is covered by `create_schema_item()`.

### create_schema_item()

85 of the 113 crawled snapshots carried Advanced protocols with schema items. Per FR-007 this factory counts only the deepest level, `Protocols[].Schemas[].SchemaItems[].Parameters`. The crawl found four distinct schema-item parameters, all clearing the D5 breadth bar. This is where the dose family lives in Advanced protocols, mirroring `create_protocol()` for the Simple case.

| argument | pksim_parameter | default_unit | dimension | occurrences | n_models | evidence |
|---|---|---|---|---|---|---|
| `dose` | `InputDose` | `mg` | Mass | 928 | 85 | D5(a)+(b): on all 85 schema-item-bearing models (100% breadth) and the core dosing constant. Resolves to the dose family by unit dimension (rule below); observed units `mg` and `mg/kg` in this corpus on the single JSON parameter `InputDose`. |
| `start_time` | `Start time` | `h` | Time | 925 | 84 | D5(a): on 84 of 85 models. Maps to the engine constant `Start time`; observed `h`/`min`/`s`/`day(s)` all Time, canonical default `h`. |
| `volume_of_water_per_body_weight` | `Volume of water/body weight` | `ml/kg` | Volume per body weight | 835 | 81 | D5(a): 81 of 85 models. Maps to the engine constant `Volume of water/body weight`; single observed unit `ml/kg` (dimension Volume per body weight). |
| `infusion_time` | `Infusion time` | `h` | Time | 78 | 24 | D5(a): 24 models (>= 5). Maps to the engine constant `Infusion time`; observed `h`/`min`/`s` all Time, canonical default `h`. |

**Dose family and unit-to-family resolution rule (FR-009, FR-011, AC-4).** As on Simple protocols, PK-Sim stores the applied dose on a schema item as a single JSON parameter named `InputDose`, and its unit dimension selects the canonical engine dose-family member. A follow-up implementing `dose` here resolves the family identically to `create_protocol()`:

- a mass unit (`mg`, dimension `Mass`) -> `Dose` / `InputDose`;
- a body-weight-normalised unit (`mg/kg`, dimension `Dose per body weight`) -> `DosePerBodyWeight`;
- a body-surface-area-normalised unit (`mg/m²`, dimension `Dose per body surface area`) -> `DosePerBodySurfaceArea`.

All three dimensions and their default units validated against `validate_unit()`. In this corpus only `mg` and `mg/kg` were observed on schema items (`mg/m²` appeared on Simple protocols), but the same dimension-driven rule covers `mg/m²` -> `DosePerBodySurfaceArea` and an amount-based `mol` -> `Dose` (dimension `Amount`) as D5(b) canonical-constant family members. The repetition parameters (`NumberOfRepetitions`, `TimeBetweenRepetitions`) live one level up on the schema and are covered by `create_schema()`.
